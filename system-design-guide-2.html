Core Infrastructure & Networking

DNS — Domain Name System
DNS is the internet's phonebook. It translates human-readable domain names (google.com) into IP addresses (142.250.80.46) so computers can locate each other.

Flow: Browser → Recursive Resolver → Root Server → TLD Server → Authoritative Server → IP returned.

Record types:
A — IPv4AAAA — IPv6CNAME — AliasMX — MailTXT — Verification
TTL (Time-To-Live): how long DNS responses are cached. Lower TTL = faster propagation but more DNS lookups.
Load Balancing
Distributes incoming traffic across multiple servers to prevent overload and ensure availability.

Algorithms:
• Round Robin — rotate through servers equally
• Least Connections — route to server with fewest active connections
• IP Hash — same client always hits same server (sticky sessions)
• Weighted — stronger servers get more traffic

Layer 4 (transport) vs Layer 7 (application, can inspect HTTP content).
Health checks: Load balancers ping servers; unhealthy ones are removed from rotation.
API Gateway
A single entry point for all client requests to backend services. Acts as a traffic cop and security guard combined.

Responsibilities:
• Route requests to correct microservice
• Authentication & Authorization
• Rate limiting & throttling
• Request/response transformation
• Logging, monitoring, analytics
• SSL termination
• Caching

Without an API Gateway, every microservice must implement auth, logging, and rate-limiting itself — massive duplication.
CDN — Content Delivery Network
A globally distributed network of edge servers that cache static content close to users, reducing latency dramatically.

What gets cached: Images, videos, CSS, JS, HTML.
How it works: User requests content → nearest edge server responds → if not cached, fetch from origin.

Benefits: Lower latency, reduced origin load, DDoS protection, higher availability.

CloudflareAWS CloudFrontFastlyAkamai
Cache invalidation is the hard part — when origin content changes, CDN caches must be purged.

Network Essentials
Latency — time for a packet to travel from A to B (ms).
Bandwidth — max data rate (Mbps/Gbps).
Throughput — actual data transferred per second.
TCP — reliable, ordered, connection-based. Uses 3-way handshake (SYN → SYN-ACK → ACK).
UDP — fast, unreliable, no connection. Good for video streaming, gaming.
HTTP/2 — multiplexing (multiple requests over one connection).
HTTP/3 — built on QUIC (UDP-based), faster connection setup, better on lossy networks.
Security Fundamentals
TLS/HTTPS — encrypts data in transit. NEVER transmit passwords in plain HTTP.
Authentication — verifying identity. Methods: Session cookies, JWT tokens, OAuth 2.0, API Keys.
Authorization — what you're allowed to do (RBAC, ABAC).
Rate Limiting — cap requests per IP/user to prevent abuse.
DDoS Protection — CDNs absorb volumetric attacks; WAF blocks application-layer attacks.
Encryption at rest — encrypt stored data (AES-256).
JWT: Header.Payload.Signature — stateless, server doesn't store sessions. But can't be invalidated before expiry without a blacklist.

ACID: Atomicity (all or nothing), Consistency (rules enforced), Isolation (concurrent txns don't interfere), Durability (committed = permanent).
Database Indexes
Indexes create a separate sorted data structure pointing to rows, so queries don't scan the entire table.

B-Tree Index — default in most SQL DBs. Great for range queries.
Hash Index — O(1) exact lookups only, no range queries.
Composite Index — index on multiple columns. Column order matters.
Covering Index — all query data in the index itself; no table scan needed.

Trade-off: Indexes speed up reads but slow down writes (must update index on insert/update/delete). Don't over-index.
Data Partitioning (Sharding)
Splitting a large database across multiple machines so no single node becomes a bottleneck.

Horizontal Sharding — split rows across nodes (most common).
Vertical Partitioning — split columns/tables across nodes.

Sharding strategies:
• Range-based — e.g., users A-M on shard1. Risk: hot spots.
• Hash-based — hash(user_id) % N. Even distribution.
• Directory-based — lookup table maps keys to shards. Flexible but single point of failure.
Resharding (adding nodes) is painful with hash-based. Consistent hashing minimizes data movement when nodes are added/removed.
Redundancy & Replication
Keeping copies of data on multiple nodes for fault tolerance and read scalability.

Master-Slave (Primary-Replica) — master handles writes, slaves replicate and serve reads. Simple but master is SPOF.
Master-Master — both nodes accept writes. Higher availability but conflict resolution is complex.
Synchronous replication — wait for replica confirmation before acknowledging write. Strong consistency, higher latency.
Asynchronous replication — write acknowledged immediately, replica updated later. Low latency, risk of data loss.
Distributed File Systems
Store large files (videos, backups, datasets) across a cluster of machines as one logical filesystem.

HDFS (Hadoop) — designed for batch processing of huge files. Files split into 128MB blocks across DataNodes. NameNode tracks metadata.
GFS (Google File System) — inspiration for HDFS.
Amazon S3 — object storage, highly durable (11 nines), infinitely scalable.

DFS are optimized for sequential reads of large files, not random access to small files.
Bloom Filters
A space-efficient probabilistic data structure that answers: "is this element in the set?"

Answers:
• "Definitely NOT in set" — 100% accurate
• "Probably in set" — may have false positives
Never false negatives.

How: Uses multiple hash functions mapping to a bit array. Set bit = element hashed there. If all bits set → probably exists.

Use cases: DB query optimization (avoid disk lookups for non-existent keys), spam filters, duplicate URL detection in crawlers.
Space: O(m bits) vs O(n keys) for hash set

Caching & Performance

Caching — Complete Picture
Storing frequently accessed data in fast storage (memory) to avoid repeated expensive computation or DB lookups.

Layers of caching:
Client-side (Browser)CDN Edge CacheApplication Cache (Redis)Database Query CacheCPU L1/L2/L3

StrategyFlowBest for
Cache-aside (Lazy)App checks cache → miss → fetch DB → update cacheRead-heavy, unpredictable access
Read-throughCache handles DB fetch automatically on missCleaner app code
Write-throughWrite to cache AND DB simultaneouslyStrong consistency
Write-backWrite to cache only → sync to DB later asyncWrite-heavy, can tolerate slight data loss
Write-aroundWrite directly to DB, bypass cacheData written once, rarely re-read

Eviction Policies:
LRU — Least Recently UsedLFU — Least Frequently UsedFIFO — First In First OutTTL — Expire after time
Cache invalidation is one of computer science's hardest problems. Two strategies: TTL (expire after time) or event-based (invalidate on write).
Cache stampede: many requests hit DB simultaneously when cache expires. Fix with mutex locks or probabilistic early expiry.

Distributed Systems Fundamentals

Key Characteristics
Properties every distributed system must balance:

Scalability — handle growing load (horizontal = more nodes, vertical = bigger node).
Reliability — work correctly even when components fail.
Availability — percentage of time system is operational (99.99% = ~52 min/year downtime).
Fault Tolerance — continue operating despite failures.
Consistency — all nodes see the same data at the same time.
Performance — low latency, high throughput.
These goals conflict. You cannot have all simultaneously — this is the essence of distributed systems design.
CAP Theorem
A distributed system can guarantee at most 2 of 3 properties during a network partition:

C — Consistency: Every read gets the most recent write.
A — Availability: Every request gets a response (not necessarily latest data).
P — Partition Tolerance: System works despite network splits.

Since partitions always occur in real networks:
• CP systems — prefer consistency over availability (HBase, ZooKeeper)
• AP systems — prefer availability over consistency (Cassandra, DynamoDB)
PACELC extends CAP: Even without partition, choose between Latency (L) vs Consistency (C). Most DBs are either PC/EC or PA/EL.
Quorum
Minimum number of nodes that must agree for an operation to be valid.

In a cluster of N nodes:
W + R > N → guarantees strong consistency
W = write quorum, R = read quorum.

Example (N=5):
• W=3, R=3 → consistent (overlapping writes and reads)
• W=1, R=5 → fast writes, slow reads
• W=5, R=1 → slow writes, fast reads
Common: W=R=⌊N/2⌋+1. Majority quorum means you can tolerate ⌊N/2⌋ node failures.
Leader & Follower
A pattern where one node (Leader/Primary) coordinates operations, others (Followers/Replicas) replicate state.

Leader responsibilities:
• Accept all writes
• Coordinate distributed transactions
• Decide ordering of operations

Follower responsibilities:
• Replicate leader's log
• Serve read requests
• Ready to become leader on failure
Leader Election is triggered when leader fails. Algorithms: Raft (easier to understand), Paxos (classic but complex), Bully algorithm.
Heartbeat
A mechanism where nodes periodically send small signals to prove they're alive.

How it works:
• Each node sends a heartbeat packet every few seconds
• Monitor (or peers) tracks last heartbeat time
• If no heartbeat for threshold duration → node declared dead
• Triggers failover, leader election, or rerouting

Used in: Kubernetes (pod health checks), ZooKeeper, distributed databases, microservices (health endpoints).
False positives: A slow network can look like a dead node. Use timeouts carefully and consider multiple missed heartbeats before declaring failure.
Checksum
A small value computed from data used to detect errors or corruption during transmission or storage.

How: Sender computes checksum of data, sends it with the data. Receiver recomputes checksum and compares. If mismatch → data corrupted.

Algorithms:
• CRC32 — fast, good for detecting random bit errors
• MD5, SHA-256 — cryptographic, also verify integrity
• Adler-32 — faster than CRC32, weaker

Used in: HDFS (every block has a checksum), TCP (header checksum), file downloads, git (SHA hash for every commit).

Message Queues vs. Service Bus
Message Queue — simple pipeline. One producer, one consumer. Message consumed once and deleted. FIFO ordering.
Use when: task distribution, background jobs.
RabbitMQAmazon SQS

Service Bus / Message Broker — routing, filtering, fan-out (pub/sub). One message can go to many consumers. Topics and subscriptions.
Use when: event notification, decoupled microservices.
Apache KafkaAzure Service Bus
Kafka is log-based — messages are retained after consumption. Multiple consumers can replay events independently. Great for event sourcing.
Synchronous vs. Asynchronous
Synchronous: Caller waits for response before continuing. Tight coupling. Simple to reason about.
Example: REST API call — you wait for HTTP response.
Problem: If downstream service is slow, you're blocked. Cascading failures.

Asynchronous: Caller continues immediately. Response comes later (callback, promise, event).
Example: Queue a background job (send email), respond "accepted 202".
Problem: Harder to debug. Need to handle out-of-order processing.
Most real systems use async for anything non-critical path: emails, notifications, analytics, image processing.
Push vs. Pull Notifications
Push: Server initiates and sends data to clients when ready. Client is passive.
✅ Real-time. ✅ No wasted polls.
❌ Server must maintain client connections. ❌ Hard to scale to millions.
Examples: Mobile push (FCM/APNS), WebSockets.

Pull: Client periodically requests data from server.
✅ Simple. ✅ Client controls timing.
❌ Polling wastes bandwidth. ❌ Delay between polls = latency.
Examples: Email clients, RSS feeds, long-polling.
Hybrid: Push to notify "new data available" → client pulls the actual data. Best of both worlds.
Event-Driven vs. Polling Architecture
Polling: Component periodically checks for changes ("any updates?").
• Simple, works anywhere
• Wasted cycles when nothing changes
• Delay = polling interval

Event-Driven: Components react to events. When something happens, emit an event; interested parties handle it.
• Real-time reaction
• Decoupled — emitter doesn't know consumers
• Highly scalable
• Harder to trace and debug (distributed traces needed)
KafkaEventBridgeSNS/SQSRabbitMQ

Architecture Patterns

Microservices vs. Serverless Architecture
AspectMicroservicesServerless
UnitService (long-running process)Function (ephemeral execution)
InfrastructureYou manage servers/containersCloud manages everything
ScalingPer-service horizontal scalingAuto-scales to zero and beyond
StateCan maintain stateStateless (state in DB/cache)
Cold startNo issueLatency spike on first invocation
CostPay for running serversPay per invocation (can be cheaper)
Best forComplex, long-running, statefulEvent-triggered, bursty, glue code
ExamplesNetflix, UberAWS Lambda, Azure Functions
Microservices challenges: Network latency between services, distributed tracing, service discovery, data consistency across services. Use API Gateway + service mesh (Istio) for production setups.
Stateful vs. Stateless Architecture
Stateless: Server remembers nothing between requests. Each request is self-contained (carries all needed context, e.g., JWT).
✅ Easy to scale horizontally (any server handles any request)
✅ Simple load balancing
✅ No sticky sessions needed
Example: REST APIs with JWT tokens.

Stateful: Server maintains session state between requests.
✅ Can maintain complex multi-step workflows
❌ Sticky sessions required (client must hit same server)
❌ Server failure = session loss
Example: Traditional session-based auth, WebSocket connections.
Best practice: Keep application servers stateless; store state externally in Redis, DB, or shared storage.

Data Processing & Formats

Batch Processing vs. Stream Processing
Batch Processing: Collect data over time → process the whole chunk at once on a schedule.
✅ Simple, efficient for large volumes
✅ Can reprocess historical data
❌ High latency (results not immediate)
Examples: Nightly payroll runs, daily report generation, ETL pipelines.
Apache HadoopSpark (batch mode)

Stream Processing: Process data continuously as it arrives, event by event.
✅ Real-time / near-real-time results
✅ Immediate reactions to events
❌ Harder to handle late-arriving data
Examples: Fraud detection, live dashboards, real-time recommendations.
Apache Kafka StreamsApache FlinkSpark Streaming
Lambda Architecture combines both: batch layer for accuracy + speed layer for low latency.

Application-Level Concepts

Full-Text Search
Searching content by meaning, not just exact key matches. Regular DB LIKE queries won't scale.

Inverted Index — the core data structure. Maps each word → list of documents containing it. Like a book's index at the back.

Process: Tokenization → Stop word removal → Stemming/Lemmatization → Index storage → Relevance scoring (TF-IDF, BM25).

Features: Fuzzy matching, synonyms, faceted search, autocomplete, geo search.
ElasticsearchSolrTypesenseMeilisearch
Architecture pattern: Write to primary DB → stream changes to Elasticsearch via CDC (Change Data Capture). Don't treat search engine as source of truth.
Notification System
Delivering messages to users across multiple channels reliably at scale.

Channels: Push (mobile), Email, SMS, In-app, WebSocket.

Fan-out problem: When a celebrity posts on Twitter, millions must be notified. Two approaches:
• Fan-out on write — pre-compute each user's feed on post. Fast reads, expensive writes.
• Fan-out on read — compute feed at read time. Slower reads, cheap writes.
• Hybrid — fan-out on write for normal users, fan-out on read for celebrities.

Key requirements: At-least-once delivery, deduplication, user preferences, rate limiting per user.
Distributed Coordination Services
Services that help distributed systems coordinate without conflicts — essentially a "source of truth" for cluster state.

Use cases:
• Leader Election — which node is master?
• Distributed Locks — prevent two nodes doing the same work
• Configuration Management — consistent config across all nodes
• Service Discovery — which IP/port is ServiceX running on?
• Membership — which nodes are alive?

ZooKeeperetcd (Kubernetes)Consul
ZooKeeper uses ZAB protocol (CP system). etcd uses Raft consensus. Both guarantee linearizability but sacrifice availability during partitions.

Monitoring & Observability
What to learn: The 3 Pillars of Observability:
• Logs — what happened (structured JSON logs preferred)
• Metrics — system numbers over time (CPU, error rate, latency)
• Traces — how a request flowed across services (distributed tracing)

Key methods: RED (Rate, Errors, Duration) for services; USE (Utilization, Saturation, Errors) for resources. SLI (metric) → SLO (target) → SLA (contract). Alerting on symptoms, not causes. OpenTelemetry is the vendor-neutral standard for all three pillars — learn it first.
